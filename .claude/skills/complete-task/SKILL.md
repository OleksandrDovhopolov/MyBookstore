# /complete-task — закрытие задачи

Назначение: единообразно фиксировать выполнение задачи в Notion и в локальном журнале `docs/PROGRESS.md`.

**Критично:** не выполнять шаги вслепую. Если есть хоть малейшая неопределённость в том, КАКАЯ именно задача закрывается или ЧТО именно было реализовано — **остановиться и спросить пользователя**. Лучше задать лишний вопрос, чем закрыть не ту задачу.

## Когда вызывается

Только когда пользователь явно пишет `/complete-task` (опционально с аргументом — названием/частью названия задачи).
НЕ срабатывает на фразы вроде "задача выполнена" в свободной форме — только явный вызов.

## Алгоритм

### 1. Определить задачу

Источники в порядке приоритета:

1. **Аргумент команды** — если пользователь написал `/complete-task <название>`, ищем по нему.
2. **Контекст разговора** — последние 5–10 сообщений: о какой задаче шла речь? Какой MD-файл из `docs/` обсуждался?
3. **Реестр** — `docs/PROGRESS.md` содержит уже закрытые задачи (не годятся для повторного закрытия). Используется для понимания формата и истории.
4. **Notion** — основной источник. Поиск через `notion-search` в базе "✅ Задачи до релиза (MVP)" (data source `collection://b569660e-d516-412e-a18e-0606d3874d68`).

Найдя кандидата, **ВСЕГДА** подтвердить с пользователем перед действиями:

> "Закрываю задачу: **<точное название из Notion>** (статус сейчас: <текущий статус>). Верно?"

Если кандидатов несколько или совпадение неточное — использовать `AskUserQuestion` со списком.

Если задача уже в статусе `Ready` / `Done` — сообщить и спросить, точно ли закрывать повторно.

### 2. Определить коммит

Источники в порядке приоритета:

1. **Аргумент** — если пользователь указал хеш.
2. **Свежий коммит** — `git log -1 --format="%h %s"`. Если он логически связан с задачей по сообщению — предложить его.
3. **Анализ git log** — последние 5 коммитов. Спросить пользователя, какой относится к задаче.

**Подтвердить:**
> "Коммит реализации: `<hash>` — <subject>. Верно?"

Если работа не закоммичена — попросить пользователя сначала закоммитить, не выдумывать хеш.

### 3. Собрать описание реализации

Из контекста разговора: что именно было сделано, какие файлы/папки созданы, какие отклонения от исходной формулировки задачи, что НЕ входило в реализацию (важно — это помогает потом понимать границы).

**Если контекста разговора недостаточно** (например, скилл вызван в новой сессии) — спросить пользователя, что было реализовано. Не придумывать.

Структура комментария (markdown):

```markdown
**Реализовано в коммите** `<hash>` (*<commit subject>*)

**Что сделано:**
- <пункт 1>
- <пункт 2>

**Что не входило в реализацию:** <если есть>

**Отклонения от исходной задачи:** <если есть>

**Документация:** <ссылки на MD в docs/, если есть>
```

### 4. Записать в Notion

1. `notion-create-comment` — добавить комментарий (markdown выше) на страницу задачи.
2. `notion-update-page` с `command: update_properties` и `properties: {"Статус": "Ready"}`.

Если поле статуса называется иначе (проверить через `notion-fetch` страницы) — использовать фактическое имя поля.

### 5. Обновить docs/PROGRESS.md

Прочитать файл, добавить новую запись в конец (после существующих) в формате:

```markdown
## <Название задачи>
- **Notion:** <url>
- **Коммит:** `<hash>` — <subject>
- **Дата:** <YYYY-MM-DD>
- **Что сделано:** <1-2 предложения>
```

Если `docs/PROGRESS.md` не существует — создать с шапкой из текущего шаблона.

### 6. Отчитаться пользователю

Короткий итог: какая задача закрыта, ссылка на Notion, какой коммит, куда добавлена запись.

## Правила безопасности

- **НЕ закрывать задачу без явного подтверждения** от пользователя на шаге 1.
- **НЕ придумывать хеш коммита.** Только из `git log`.
- **НЕ придумывать содержание реализации.** Только из контекста разговора или ответов пользователя.
- **НЕ перезаписывать существующие записи в PROGRESS.md** — только добавлять новые.
- Если задача в статусе `Ready`/`Done` — переспросить.
- Если статус не "Ready" в схеме Notion (например, "Готово" / "Done") — использовать фактическое значение из data source schema.
