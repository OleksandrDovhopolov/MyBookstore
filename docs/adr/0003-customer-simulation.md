# ADR-0003: Customer simulation — фаза «Продажа» как real-time симуляция

- **Status:** Accepted
- **Date:** 2026-06-11
- **Deciders:** project owner
- **Supersedes:** значительную часть Sales MVP slice (пошаговый `SalesSessionService` как оркестратор)
- **Related:** `docs/INPROGRESS/Продажа.md`, `docs/INPROGRESS/SalesFeatureImplementationPlan.md`, `docs/INPROGRESS/customer-simulation-decisions.md`, `docs/REFERENCE_TinyBookshop_Day.md`, [ADR-0001](0001-save-data-modular-payload.md)

## Context

Текущая фаза «Продажа» (`Assets/Game/Features/BookSell/`) реализована как **пошаговая модель**: очередь из N `RequestConfig`, один активный запрос в момент времени, игрок обязан закрыть его (`RecommendBook`/`Skip`), пассивные продажи — сайд-эффект между ходами. Нет времени, движения, одновременности. Домен чистый и покрыт EditMode-тестами.

Референс (Tiny Bookshop, см. `docs/REFERENCE_TinyBookshop_Day.md`): покупатели **приходят** к лавке, **бродят**, покупают **N книг** сами (пассивно), часть просит **активную помощь** (миниигра подбора). Чтобы приблизиться к этому core-ощущению, фазу нужно превратить в **real-time симуляцию конкурентных агентов-покупателей**.

Ограничения, формировавшие решение:

- **Соло-игра.** Соревновательность / лидерборды / события / PvP — не планируются в принципе. Threat model — только накрутка ресурсов (gold/репутация) одиночным игроком.
- MVP, один разработчик. Цель прототипа — проверить «интересно ли играть 5 дней».
- Существующий домен Sales (скоринг, пассивный селектор) — чистый, тестируемый; его нужно сохранить и переиспользовать.
- Строгие asmdef-правила; Unity 6, VContainer, UniTask, Newtonsoft.
- ADR-0001: «No shared C# code between client and server is planned»; server-authoritative logic — триггер пересмотра.

Зафиксированные владельцем решения по поведению (полностью — в `customer-simulation-decisions.md`): унифицированный агент (intent'ы passive/active в любом порядке, активный не обязан следовать за пассивным); общий FIFO-lock на миниигру+диалог; пауза замораживает чужие тики; reserve-on-target; вместимость лавки не нужна; concurrency бесконечно; спавн за интерфейсом; конец дня = все обслужены ИЛИ книги кончились; miss пассивки пропускает книгу; save во время дня пересоздаётся.

Рассмотренные и **отклонённые** подходы к рантайму (детально — ниже в «Что НЕ выбрано»): паттерн `Game.Commands.ICommand`, серверный deterministic-replay, коммерческий Behavior Designer.

## Decision

Реализовать симуляцию покупателей как **tick-driven state machine, управляемую персональным планом (Plan) каждого покупателя**, с разделением «мозг/тело» и внешним арбитражом общих ресурсов.

### 1. Customer = Plan (данные) + tick-FSM (исполнитель)

- **Plan** — упорядоченный список шагов (`IReadOnlyList<ICustomerStep>`): `Approach → Browse → {passive|active}* → (опц. Dialogue) → Leave`. Это **данные**, которые выдаёт спавнер.
- **Шаг** — резюмируемое состояние, не одноразовый `Execute()`:
  ```
  interface ICustomerStep { void Enter(ctx); StepStatus Tick(ctx, dt); void Exit(ctx); }
  enum StepStatus { Running, Completed, Blocked }   // Blocked = жду lock / ввод игрока
  ```
- **Customer** (brain) держит план + индекс текущего шага; `Tick(dt)` продвигает автомат.
- Шаги MVP: `ApproachStep(duration)`, `PassivePurchaseStep`, `ActiveRequestStep`, `DialogueStep`, `LeaveStep`.

### 2. Brain / View split

- **Brain (домен):** чистый C#, **без Unity-типов и без wall-clock**. «Движение A→B» = доменная длительность `ApproachDuration` + состояние `Approaching` (без координат). Полностью EditMode-тестируем.
- **View (presentation):** `MonoBehaviour`, реальное движение/анимации/попапы/клик по покупателю. Подписан на события мозга (`StepEntered`, `BookTargeted`, `BookBought`, `BookRejected`, `ShoppingFinished`) и проигрывает визуал.

### 3. Унифицированный агент

Покупатель = последовательность intent'ов покупки (каждый — passive или active, в любом порядке) + опциональный диалог. Допустим один active-intent на одну книгу без пассива. После активного intent'а — возможен возврат в пассивный режим. «Active/passive» — режимы поведения, не разные классы.

### 4. Общий арбитраж — `IInteractionLock`

- **Один** lock на (миниигра ∪ диалог). Диалог и миниигра не идут одновременно.
- **FIFO**-очередь ожидающих. Только 1 попап запроса помощи в момент времени.
- Триггер consume — **подменяемая стратегия**: сейчас авто, позже — клик по UI.
- Шаг `ActiveRequestStep`/`DialogueStep` при занятом lock возвращает `Blocked`.

### 5. Доменная пауза (не `Time.timeScale`)

Контроллер при удержанном lock **не вызывает `Tick`** у остальных покупателей и у спавнера → доменные таймеры замирают, экономика не меняется. Анимации/твины/UI/звук (presentation) продолжают играть. Требует единого `Tick(dt)` в контроллере, а не разрозненных `await UniTask.Delay` по агентам.

### 6. Reserve-on-target (контерженция «1 экземпляр книги»)

Как только покупатель «нацелился» на книгу — она резервируется (soft-lock) и **исчезает с полки активной миниигры**. Запауженный покупатель удерживает резерв. Пассивный селектор фильтрует уже зарезервированные.

### 7. Спавн за интерфейсом

`ICustomerSpawner` — pluggable (поведение часто меняется, генерируется отдельной системой). Placeholder: последовательно 1-за-1. Concurrency не ограничена; покупатель о количестве не знает.

### 8. Day controller

`SalesDayController` (заменяет оркестраторную роль `SalesSessionService`): тикает покупателей, владеет `IInteractionLock`, спавнером, полкой/резервами; завершает день, когда **обслужены все покупатели ИЛИ распроданы все книги**. Переиспеользует без изменений: `IRecommendationScoringService`, `IPassiveSaleSelector` (+ фильтр резервов), `ISalesRandom`, конфиги.

### 9. Без server-replay и без `ICommand`

Серверный пересчёт дня не делаем. Анти-чит **отложен до релиза**; направление — server-side bounds-валидация **итога дня** (`SalesDayResult`) на фазе Итогов, не пересчёт входов. Детерминизм нужен только ради юнит-тестов (seeded `ISalesRandom`, фиксированный `dt`), не cross-platform.

### 10. Save

Во время дня не сохраняется; рестарт пересоздаёт день. Сворачивание/закрытие приложения посреди дня → день теряется (MVP-поведение, см. open item).

## Consequences

### Positive

- **Тестируемость сохранена.** Brain — чистый домен; весь флоу покупателя (планы, покупки, lock, резервы) покрывается EditMode без сцены, как сейчас покрыт скоринг.
- **Пауза и одновременность — бесплатно.** Пауза = «не тикать»; concurrency = независимые автоматы в цикле контроллера. Никаких фейковых wait-команд и борьбы с `Time.timeScale`.
- **Переиспользование.** Скоринг/селектор/random/конфиги остаются как есть — меняется только оркестратор.
- **План — это данные.** Спавнер задаёт поведение покупателя списком шагов; легко варьировать состав (сколько пассивных, есть ли активный/диалог) без новых классов.
- **Ноль внешних зависимостей и стоимости.** Свой ~маленький FSM (~150–300 строк) вместо коммерческого фреймворка; полностью под контролем.
- **Расширяемость без переписывания.** Линейный план с ветвлениями естественно дорастает до Behavior Tree (pure-C#), если поведение усложнится.

### Negative

- **Больше движущихся частей**, чем у пошаговой модели: lock, резервы, спавнер, tick-цикл, brain/view-события. Сложность сместилась с «одного хода» на «арбитраж между агентами».
- **Рефактор существующего Sales.** Оркестраторная роль `SalesSessionService` уходит в `SalesDayController`; часть текущих тестов (`SalesSessionServiceTests`) переписывается под новый контракт. Скоринг/селектор-тесты остаются.
- **Real-time вводит тайминговые нюансы** (порядок событий в тике, момент резерва vs покупки) — требуют аккуратных тестов на гонки внутри одного тика.
- **Detерминизм — ответственность домена.** `UnityEngine.Random` в `UnityRandomSalesRandom` не детерминирован между запусками; для воспроизводимых тестов нужен seeded PRNG (для прода допустим текущий).
- **Save-gap.** Потеря дня при сворачивании на мобиле может раздражать (см. open item) — осознанный MVP-компромисс.

### Что НЕ выбрано и почему

1. **`Game.Commands.ICommand` как рантайм симуляции.** Поведенческий one-shot (`Execute()`→`Completed`), Unity-bound (`Time.realtimeSinceStartup`), несериализуемый, со статическим `_executingCache`. Не умеет пауза / ожидание общего ресурса / реакция на ввод в середине / удержание резерва — это **состояние во времени**, а не «выстрелил и забыл». Импеданс-мисматч.
2. **Серверный deterministic-replay дня (command sourcing).** Требует детерминированного ядра, своего PRNG, float-дисциплины, детерминированного спавна, общего/дублированного ядра на сервере. Для соло-игры с угрозой «накрутка ресурсов» — оверкилл с ≈нулевой отдачей (клиентский чит неостановим в принципе, никто кроме читера не страдает). Противоречит ADR-0001.
3. **Behavior Designer (коммерческий BT).** GameObject-bound рантайм ломает brain/view split и тестируемость; пауза дерётся с Unity-loop; наша логика слишком проста для BT; и главное — BD **не решает** реальную сложность (lock, резерв, пауза, спавн — это арбитраж между агентами, вне дерева одного агента). Отрицательный ROI.
4. **`Time.timeScale = 0` для паузы.** Сломает UI-твины, анимации попапов, звук. Пауза должна быть доменной.

## Extension paths and complexity

Расширение — через новые шаги / новые реализации интерфейсов, без правки ядра.

### 1. Конечное терпение покупателя
**Что делать:** добавить таймер в `WaitingForLock`/`Browsing`; по истечении — переход в `LeaveStep` с пометкой «ушёл недовольным». Хук уже заложен (терпение = ∞ по умолчанию).
**Сложность:** **Low (~0.5 дня)**.

### 2. Уникальные персонажи + квесты/диалоги
**Что делать:** `DialogueStep` уже в плане; добавить источник диалога (квест-сервис) и спец-флаг покупателя в `ICustomerSpawner`. Диалог делит общий lock.
**Сложность:** **Medium** — зависит от диалоговой системы, не от симуляции.

### 3. Настроение / репутация, влияющие на покупки
**Что делать:** доменные модификаторы в `PassivePurchaseStep`/скоринге; чистая логика, тестируется.
**Сложность:** **Low–Medium**.

### 4. Клик-по-покупателю вместо авто-триггера
**Что делать:** подменить стратегию consume у `IInteractionLock` (auto → player-click). Точка расширения заложена (решение #3 decisions-doc).
**Сложность:** **Low**, плюс UI.

### 5. Setup из фазы Подготовки + спавн из DayConfig
**Что делать:** `SalesSessionSetup` уже принимается; `ICustomerSpawner` параметризовать `DayConfig` (события → больше покупателей определённого типа).
**Сложность:** **Low** на стороне симуляции.

### 6. Декор, влияющий на пассивные/активные продажи
**Что делать:** `DecorIds` уже хранятся в `SalesSessionSetup`; подключить к скорингу/селектору как модификаторы.
**Сложность:** **Low–Medium**.

### 7. Полноценное Behavior Tree
**Что делать:** если ветвлений станет много — заменить линейный Plan на pure-C# BT (узлы Sequence/Selector/Decorator), сохранив brain/view split. **Не** коммерческий GameObject-bound ассет.
**Сложность:** **Medium–High**, но только когда реально понадобится.

### 8. Save во время дня
**Что делать:** сериализовать состояние контроллера (покупатели + их шаги + резервы + lock) в модуль `book_sell.sales`. Усложняется reserve/lock-состоянием.
**Сложность:** **Medium–High**. Сейчас сознательно избегаем.

### 9. Server-side bounds-валидация (анти-чит)
**Что делать:** на фазе Итогов сервер проверяет дельту дня (`gold_day ≤ customers × max_book_price × k`, монотонный day index, без отрицательных/скачков). Нужен только `SalesDayResult`. Не на стороне симуляции.
**Сложность:** **Medium** (в основном серверная).

## When to revisit

Этот ADR стоит пересмотреть, если:

- **Поведение покупателей станет глубоко ветвящимся** (приоритеты, прерывания, перепланирование, социалка между агентами) — линейный Plan перерастает в Behavior Tree (см. §7).
- **Появится соревновательность / лидерборды / события** — отменяет threat model «соло-накрутка», поднимает вопрос server-authoritative логики и пересмотра ADR-0001 (тогда — replay/серверный авторитет имеют смысл).
- **Понадобится восстановление дня** после сворачивания/закрытия — потребуется save во время дня (§8).
- **Производительность при многих покупателях** (десятки одновременных) станет проблемой — пересмотреть tick-цикл / пулинг view / аллокации.
- **Детерминизм понадобится cross-platform** (маловероятно без п.2) — заменить PRNG и вынести всю математику в целочисленную/decimal.

## Implementation notes

- Реализация — в существующей сборке `Book.Sell` (`Assets/Game/Features/BookSell/`); симуляция — эволюция Sales, не новая фича.
- Слои: brain/домен (`Domain/`, `Services/` — чистый C#), view (`UI/` — `MonoBehaviour`), контроллер (`SalesDayController`), арбитраж (`IInteractionLock`), спавн (`ICustomerSpawner`).
- Переиспользуются без изменений: `IRecommendationScoringService`, `IPassiveSaleSelector` (+ фильтр резервов), `ISalesRandom`, `BookConfig`/`RequestConfig`/`LocationConfig`/`RequestDifficulty`.
- `SalesSessionService` (пошаговый) — заменяется `SalesDayController`; миграция: сохранить скоринг/селектор-тесты, переписать session-тесты под tick-модель.
- DI-регистрация — `BookSellVContainerBindings.RegisterBookSell()` в **`LocationLifetimeScope`** (LocationScene, additive — см. [GameFlowLoop.md](../GameFlowLoop.md)); включает `IInteractionLock`, `ICustomerSpawner`, `SalesDayController`. `ISalesShelfStateService` вынесен **глобально** (`RegisterBookSellSharedState` в `BootstrapInstaller`), т.к. общий с хабом (Preparation).
- Все строки/комментарии в коде — английские (LANGUAGE_POLICY). Контент конфигов — отдельно.
- Детальные поведенческие решения — `docs/INPROGRESS/customer-simulation-decisions.md` (источник истины по правилам; этот ADR — архитектурное решение).
