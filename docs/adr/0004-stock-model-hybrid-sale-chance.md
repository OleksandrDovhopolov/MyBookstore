# ADR-0004: Stock model (hybrid) + passive sale chance

- **Status:** Accepted
- **Date:** 2026-06-12
- **Deciders:** project owner
- **Supersedes:** уточняет пассивную часть [ADR-0003](0003-customer-simulation.md) (детерминированный матч → вероятностный sale chance)
- **Related:** [ADR-0003](0003-customer-simulation.md), `docs/INPROGRESS/FTUE — первый вход.md`, `docs/INPROGRESS/Продажа.md`, `docs/INPROGRESS/Подготовка.md`, `docs/REFERENCE_TinyBookshop_Day.md`, Notion-задача «Sales — пассивные продажи через baseSaleChance»

## Context

Нужно зафиксировать **модель стока книг** до реализации `baseSaleChance` и «Подготовки» — обе блокированы этим решением.

Что показал референс (TB) и анализ FTUE (`docs/INPROGRESS/FTUE — первый вход.md`):
- Сток в TB отображается **счётчиками по жанрам** (Crime: 7, Fantasy: 9, …); правый панель + hover = **Sale Chance** жанра.
- Туториал TB: «Sale Chance зависит от того, сколько книг в магазине, и **падает по мере распродажи**».
- Стокинг в «Подготовке» — «выбери жанр → заполни полку из склада», вместимость `In Shop X/40`; есть **storage/shop split**.
- Но **активная** рекомендация (миниигра с обложкой/блёрбом) — про **конкретную титульную книгу**.
- Декор в TB — модификатор «+X% к шансу <категория>-продажи».

Текущий код (ADR-0003): **полный per-title** сток (`SalesShelf` = список `ShelfBook`/`BookConfig`), пассив — **детерминированный** матч по `LocationConfig.DemandGenres/Tags`, без вероятности; декор игнорируется (`DecorIds` пробрасываются вхолостую).

Ограничения: соло-игра, MVP, переиспользовать существующий per-title контент + скоринг/селектор; контент только вымышленный (без real IP).

## Decision

Принять **гибридную** модель (TB-faithful).

### 1. Инвентарь = per-title; genre-count = производный агрегат
- Книга остаётся **конкретным титульным экземпляром** (`BookConfig`: Title, Author, Genre, Tags, Mood, BasePrice, RarityWeight). Контент **вымышленный**.
- «Счётчик жанра» (для правой панели и sale chance) — **агрегат** (group-by Genre над книгами склада/полки), а не отдельная сущность. Per-title сток сохраняется; genre-count надстраивается сверху.

### 2. Активная рекомендация = per-title (без изменений)
- Игрок выбирает **конкретную** книгу, видит полную инфу (название/автор/блёрб/жанр/теги/тон/цена), глубокий матчинг через существующий `RecommendationScoringService` + `RecommendationReason`. Здесь живёт вся глубина тегов/тона.

### 3. Пассивная продажа = двухстадийная вероятностная (supersedes ADR-0003 детерминизм)

**Стадия 1 — ворота по жанру (вероятность):**
- У пассивного покупателя есть **предпочитаемые жанры** (FTUE: «люблю Classic + Crime»).
- Для целевого жанра (из `liked genres` ∩ наличие на полке):
  ```
  saleChance(genre) = f(count(genre)) × locationMod(genre) × decorMod(genre)
  ```
  - `f(count)` — растёт с числом книг жанра, **падает по мере распродажи** (правило TB); конкретная формула — в задаче `baseSaleChance`.
  - `locationMod` — спрос локации бустит «свои» жанры (**модификатор**, не жёсткий фильтр).
  - `decorMod` — декор «+X% к шансу категории» (ради чего декор и существует).
- Roll. Не прошёл → продажи нет (FTUE: «Classic fail»). Это завершает passive-цепочку текущего покупателя: новых passive-attempt'ов после такого miss нет, но дальше у него ещё могут быть non-passive шаги (active request / comment / dialogue / etc.).

**Стадия 2 — какая книга уходит: взвешенный рандом по `RarityWeight`:**
- На успехе — выбрать конкретную книгу жанра с полки, **взвешенно по `RarityWeight`** (частые/дешёвые уходят первыми, редкие/дорогие — последними).
- Продать: `count(genre)--`, `+book.BasePrice`, флейвор «I read about <Title>».
- Пассив **НЕ** использует матч по тегам/тону — это слой активной миниигры (иначе пассив и актив сливаются).

### 4. Genre-count поверхности
- Правая панель — счётчики жанров; hover → sale chance жанра.
- Storage/shop split + вместимость (`dailyBookSlots`, «Подготовка») оперируют книгами, сгруппированными по жанру.

## Consequences

### Positive
- **TB-faithful** на обоих слоях: счётчики/шанс для пассива, титулы для актива.
- **Переиспользует** per-title контент + существующее поле `RarityWeight` (стадия 2) + скоринг/селектор (актив).
- **Аддитивная** правка, а не переписывание: per-title сток остаётся, genre-count — агрегат.
- Декор **наконец осмыслен** (модификатор sale chance).
- Sale chance получает конкретную форму (`f(count) × loc × decor`), глубина актива сохранена.
- «Продать последнюю книгу жанра — мастерство» (фраза TB) выпадает из взвешенного рандома естественно.

### Negative
- Два представления (per-title список ⟷ genre-count) надо держать согласованными — риск низкий, т.к. count **производный** (один источник истины — список книг).
- Пассив теперь со случайностью → нужен seeded `ISalesRandom` для детерминизма тестов.
- `f(count)` + модификаторы требуют **балансировки**.
- Supersedes детерминированный пассив ADR-0003 → переделать `PassivePurchaseStep` + `IPassiveSaleSelector` + их тесты.
- Нужен `DecorConfig` (модификаторы по категориям) — раньше декор был заглушкой.

### Что НЕ выбрано и почему
1. **Полный per-genre-count** (без титулов даже в активе) — теряется «душа» рекомендации конкретной книги (сердце фантазии книготорговца).
2. **Чистый рандом на стадии 2** — отвергнут в пользу взвешенного: пер-чистый не даёт фактуры «редкие уходят последними» и не задаёт ценность редких книг.
3. **Матч по тегам/тону в пассиве** — дублирует активный слой, размывает грань пассив/актив, дороже.

## Extension paths and complexity

1. **Скрытые предпочтения покупателя глубже жанра** (теги/тон как soft-вес на стадии 2). **Low–Medium** — добавить вес к взвешенному рандому, не меняя ворота.
2. **Ценность редких книг / коллекция** — `RarityWeight` уже задаёт «последние уходят труднее»; можно навесить отдельную цену/спрос на редкие. **Medium**.
3. **Варианты формулы `f(count)`** (линейная / `1-(1-p)^count` / с потолком) — это балансная настройка в задаче `baseSaleChance`. **Low**.
4. **Сет-бонусы декора** (как TB-теги maritime/classy) — расширение `decorMod`. **Medium**.
5. **Локации с разным базовым спросом/анлоком** — `locationMod` + карта/visit cost (FTUE). **Medium**.

## When to revisit
- Если авторинг per-title контента станет слишком тяжёлым → можно схлопнуть актив до «репрезентативных титулов» на жанр.
- Если пассив потребует больше глубины → ввести теги как **мягкий** вес на стадии 2 (не ворота).
- Если появится соревновательность/сервер-авторитет (см. ADR-0003) → вероятностный пассив потребует seed-фиксации (сейчас не нужно).

## Implementation notes
- **Реализуется** в рамках Notion-задачи «Sales — пассивные продажи через baseSaleChance»; этот ADR — решение.
- `BookConfig.RarityWeight` — вес для стадии 2 (уже есть).
- Новое: genre-count агрегат над `SalesShelf`; расчёт `saleChance` (`f(count) × locationMod × decorMod`); двухстадийный `PassivePurchaseStep`; `DecorConfig` с модификаторами по категориям.
- `ISalesRandom` — обе стадии (seeded в тестах).
- Покупателю/спавнеру добавить **liked genres** (пассив таргетит предпочитаемые жанры).
- Storage/shop split + capacity — часть «Подготовки» (`docs/INPROGRESS/Подготовка.md`), оперируют genre-grouped книгами.
