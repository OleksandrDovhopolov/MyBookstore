DI: `UiSystemVContainerBindings.RegisterUiSystem(UICanvasRoot prefab)` registers the prefab via `RegisterComponentInNewPrefab(...).UnderTransform((Transform)null)` and forces eager initialization with `RegisterBuildCallback(resolver => resolver.Resolve<IUIManager>())` so the canvas spawns at boot instead of lazily on first Show; `BootstrapInstaller` carries the prefab via `[SerializeField]`. Phase 1 pilot windows validate the architecture under real interaction: `SettingsWindow` (Page, Main, `keepInCache: true`) increments an open-count label on each `OnShowStart` to validate the cache mechanic; `ConfirmDialog` (Popup, Additional) implements `IResultWindow<ConfirmDialogResult>` with `Title`/`Body`/`ConfirmLabel`/`CancelLabel` args and is `await`-ed via `WaitForResultAsync<ConfirmDialogResult>()`; `WindowController` got `[Inject] IUIManager` + `protected CloseAsync()` helper (controllers close themselves on button clicks); `UIManager.HideAsync(IWindowController, ...)` overload; `UiPilotDebugPanel` (MonoBehaviour with OnGUI, behind `#if UNITY_EDITOR || DEVELOPMENT_BUILD`) lives on `UIManagerCanvas` and offers Show Settings / Show Confirm / Hide Top buttons. Unity setup is captured procedurally in the consolidated reference. Out of scope for this slice — and tracked as a single follow-up Notion task — ShopWindow, NoInternetWindow + ConnectionService bridge, HUD migration (GameHudView → HudRoot), migration of existing scene-placed views (Morning/Preparation/Sales/Results/Inventory), `TransitionAnimationService` (dim/fade between scenes), Props-as-SO, MVVM controller variant, MessagePipe UI events, pre-warm for frequently-opened windows, dimmed overlay under Additional, Develop-layer debug windows. ## Resources module (soft currency)
- **Notion:** https://app.notion.com/p/370511859db3814eac84eadbe3edc766
- **Commit:** `0041a81` — add resources
- **Date:** 2026-06-14
- **Summary:** Generic id-driven player wallet with a forward-compat seam for hard currency, plus a sibling Progression feature for reputation. Two new `Assets/Game/Features/` modules with API/impl asmdef split each — `Game.Resources(.API)` and `Game.Progression(.API)` — mirroring the Inventory pattern. `IResourcesService` exposes sync read (`GetAmount(id)`, `Has(id, amount)`, `GetAll()`) against an in-memory cache and async writes (`AddAsync(id, amount, reason, ct)`, `RemoveAsync(...)`) through `IResourcesRepository`; mutations fire a signed `Changed` event with audit `reason` (`"ftue"`, `"results_day_3"`, etc.). Semantics: non-positive amounts are no-ops, RemoveAsync on insufficient balance returns false without state change, remove-to-zero keeps the entry (so UI can render 0). Load happens via `ISaveHook.AfterLoadAsync` — the service self-registers in its constructor; `Bootstrap.cs` injects `IResourcesService` and `IProgressionService` to force construction before `SaveDataLoadOperation`. Persistence: `SaveBackedResourcesRepository` over `ISaveService` save module `"resources"` (schema 1), storing `ResourcesStateDto { Dictionary<string,int> Amounts }`. Server-readiness seam: a future `HttpResourcesRepository` swaps in via a single DI line. Constants `ResourceIds.{Gold, Gems}` ship as forward-compat (gems has no special behaviour — the generic API handles both identically). `Game.Progression` is the twin module for reputation — `IProgressionService.Reputation { get; }` + `AddReputationAsync(delta, reason, ct)` with a ≥ 0 clamp that reports the effective delta (so a -10 hit on a Reputation of 2 fires `Delta = -2`, not -10). Same save-hook lifecycle, save module `"progression"`. Debug HUD: `GameHudView` (TMP labels `Gold: N / Rep: M`) is injected from `GameplayLifetimeScope` as an AutoInjectedGameObject — registered there, NOT in `ResourcesVContainerBindings`, because the view lives in GameplayScene and the binding fires in boot. **`DayProgressState.Gold` and `DayProgressState.Reputation` removed entirely** — `FtueBootstrapper` now seeds 60 gold via `_resources.AddAsync(ResourceIds.Gold, 60, "ftue", ct)`; `ResultsSessionService` reads `_progression.Reputation`, applies deltas through `_resources.AddAsync(gold, ...)` + `_progression.AddReputationAsync(...)` and drops the now-pointless `_dayProgress.SaveAsync(ct)` because the `day_progress` module is no longer touched on apply. `ResultsSessionServiceTests` migrated to `FakeResourcesService` + `FakeProgressionService` (new fakes in `DayCycle.Tests.Editor.Fakes`). DI: `ResourcesVContainerBindings.RegisterResources()` and `ProgressionVContainerBindings.RegisterProgression()` are called from `BootstrapInstaller` (GlobalLifetimeScope) so FTUE in the boot scene and Results in GameplayScene resolve the same instances; previous placeholder calls in `GameInstaller` were removed to avoid double-registration. EditMode tests in two new asmdefs: `Game.Resources.Tests.Editor` (`ResourcesServiceTests`, 10 cases — self-registration as hook, add/accumulate/no-op, remove sufficient/insufficient/all-the-way, GetAll, roundtrip, unknown id) and `Game.Progression.Tests.Editor` (`ProgressionServiceTests`, 6 cases — positive/negative/beyond-zero clamp/zero no-op/roundtrip). Bonus documentation: `docs/TODO_SAVE_HOOK_BOOTSTRAP.md` proposes a future `SaveHookBootstrapper : IStartable` to replace the forced-construction injection pattern in `Bootstrap.cs`; `docs/improvements/RESOURCE_IDS_OPEN_CLOSED.md` outlines a hybrid `ResourceConfig` + `IResourceRegistry` migration to drop the centralised `ResourceIds` constants (deferred until a second currency lands). Deferred: production HUD, `HttpResourcesRepository` (server validation for hard currency), `ResourceConfig` metadata (icon/display name/IsHardCurrency/max cap), Reputation-to-Location unlock checks, hard caps, replenishable resources. Deviations from the original task: forward-compat `Gems` constant was added even though the task said "Hard currency в MVP не делаем" (only an id, no behaviour); a parallel Progression feature shipped in the same commit (separate concern but bundled); `DayProgressState.Gold/Reputation` fields were removed (not in the original task — mirrors the `OwnedBookIds` removal from Inventory).

## Sales — passive sales via baseSaleChance (probabilistic gate + decor seam)
- **Notion:** https://app.notion.com/p/37c511859db381bf89e6f89b7e1ce95a
- **Commit:** `b85e91c` — update to sell chance
- **Date:** 2026-06-17
- **Summary:** Replaced the deterministic passive-sale model with a two-stage probabilistic gate (ADR-0004). New `IBaseSaleChanceCalculator` + `EconomyBasedSaleChanceCalculator` compute per-genre chance as `clamp(min(BaseSaleChance + PerCopyChance * count, CapChance) * locMod * decorMod, 0, 1)` with parameters centralised in `EconomyConfig` (singleton); `LocationDemandMultiplier` applies when the genre appears in `LocationConfig.DemandGenres`. New `Book.Sell.API/IDecorModifierProvider` seam supplies `GetGenreMultiplier(genre, activeDecorIds)` — `SalesSessionSetup.DecorIds` are no longer ignored. `WeightedPassiveSaleSelector` rolls per genre on each tick via `ISalesRandom.NextDouble()`, picks one winning genre uniformly, then picks a specific title with cumulative-sum walk weighted by `BookConfig.RarityWeight` (with `1e-4f` floor for non-positive weights); miss-by-chance is now a distinct outcome (no sale even when demand matches). `PassivePurchaseStep` updated to roll-before-commit. EditMode tests: `WeightedPassiveSaleSelectorTests`, `EconomyBasedSaleChanceCalculatorTests`, refreshed `PassivePurchaseStepTests` — determinism via `FakeSalesRandom` + `FakeBaseSaleChanceCalculator`. Per ADR-0004 and `CORE_LOOP.md` §3.1: `baseSaleChance` lives per-genre in `EconomyConfig` (not on `BookConfig`), and passive sales intentionally ignore tags/mood — that depth belongs to the active mini-game. Deferred: real `DecorConfig` coefficients (the seam is wired but values are stubbed until the decor feature ships — see `CORE_LOOP.md` §2.3) and the sale-chance hover indicator on the production sales UI.
