> Контекст обсуждения: расширение фазы «Продажа» из пошаговой очереди запросов
> в real-time симуляцию покупателей. Спека: `docs/INPROGRESS/Продажа.md`,
> текущая реализация: `Assets/Game/Features/BookSell/`.

## Парадигма

Переход от текущей **пошаговой** модели (`SalesSessionService`: очередь из N `RequestConfig`,
один запрос в момент времени, нет времени/движения/одновременности) к **real-time симуляции
конкурентных агентов-покупателей**.

Переиспользуется: `IRecommendationScoringService`, `IPassiveSaleSelector`, `ISalesRandom`, конфиги.
Меняется оркестратор. ADR пометит, что он **supersedes** значительную часть Sales MVP slice
и пересматривает пункты `Продажа.md` → Out of Scope «свободное перемещение» и «patience simulation».

## Унифицированная модель покупателя

**Покупатель = последовательность intent'ов покупки + опц. диалог (для уникальных персонажей).**

- Каждый intent — это либо **passive** (авто-резолв через `IPassiveSaleSelector`),
  либо **active** (триггерит миниигру выбора книги, скоринг через `IRecommendationScoringService`).
- Порядок и состав произвольны. **Активный не обязан идти после пассивного.**
  Допустимо: ровно 1 intent = 1 активный запрос на 1 книгу, без пассива вообще.
- «Active» и «passive» — **режимы поведения одного агента**, не разные классы.
- После активного intent'а покупатель **может вернуться в пассивный** режим и докупить ещё
  (решение 3.3.5 — разрешено).

## Зафиксированные решения

| # | Тема | Решение |
|---|---|---|
| 1 | Пауза пассивного при чужой миниигре/диалоге | **(a) Замирает на текущем шаге** — таймер раздумья на паузе. Ничего, что меняет экономику, не происходит. |
| 2 | Очередь активных запросов | **FIFO.** Только **1 попап запроса помощи** в момент времени. |
| 3 | Кто инициирует активный запрос | **Сейчас — авто.** Когда появится UI — ждёт клика по UI. Триггер должен быть подменяемым (auto ↔ click). |
| 4 | Контерженция «1 экземпляр книги» | **Reserve-on-target.** Как только покупатель «нацелился» на книгу — она резервируется (soft-lock). Зарезервированную книгу **активный игрок уже не видит** на полке миниигры. Запауженный покупатель удерживает резерв. |
| 5 | Вместимость лавки | **Не нужна.** Покупатель ничего не знает о числе одновременных. 5–15 покупателей в день. |
| 6 | Терпение / уход по тайм-ауту | **Хук на бесконечность.** Сейчас покупатель ждёт lock бесконечно; заложить точку расширения под конечное терпение. |
| 7 | Конец дня | День заканчивается, когда **обслужены все N покупателей** ИЛИ **закончились книги** (распроданы). |
| 8 | Max concurrent customers | **Бесконечно.** Сущность покупателя об этом не знает. |
| 9 | Lock миниигры и диалога | **Один общий lock.** Диалог и миниигра не могут идти одновременно. |
| 10 | Спавн-расписание | **За интерфейсом** (поведение часто меняется, генерируется отдельной системой). Текущий placeholder: **1 за 1** (последовательно). |
| 11 | Miss в пассивке | **Пропускает книгу и продолжает** шопинг (не завершает весь визит). |
| 12 | Save во время дня | **Пока пересоздаётся.** День не восстанавливается. |

## Архитектурный водораздел (рекомендация, в ADR)

- **Customer brain** (домен, чистый, tick-driven, тестируемый): state machine, список intent'ов,
  решения «какую книгу пробовать / купил / не купил», interaction-lock, резерв инвентаря.
  **Не знает** про координаты/спрайты/твины. «Движение A→B» = доменная длительность
  `ApproachDuration` + состояние `Approaching`, без координат.
- **Customer view** (presentation, MonoBehaviour): реальное движение, анимации попапов (раздел 6
  обсуждения), клик по покупателю. Подписывается на события мозга и проигрывает визуал.
- **Спавн/генерация покупателей** — отдельная система за интерфейсом (решение #10).
  Симуляция работает с отдельным покупателем + общим lock; о количестве/частоте не знает.

## Пауза

Доменная пауза, **не** `Time.timeScale = 0` (иначе ломаются UI-твины/анимации/звук).
Глобальный флаг lock'а уважают только тики, меняющие игровое состояние:
доменные таймеры (`Approaching`/`Browsing`/`PurchaseAttempt`) и спавн — замирают;
анимации/твины/UI — продолжают. Требует единого доменного `tick(deltaTime)`,
а не разрозненных `await UniTask.Delay` по агентам.

## State machine (черновик)

```
Spawning → Approaching → Browsing
                           ├─ (passive) PurchaseAttempt → Bought/Skipped(miss) → next intent / Leaving
                           ├─ (active)  WaitingForLock → ActiveMinigame → resolve → (опц.) Browsing
                           └─ (dialogue) WaitingForLock → Dialogue → resolve → Browsing
        → WaitingForLock (терпение = ∞, хук)
        → Leaving → Despawned
```

## Серверная верификация / анти-чит — РЕШЕНО (отбросить replay)

- **Threat model:** только накрутка ресурсов (gold/репутация) одиночным игроком.
  Соревновательность / лидерборды / события / PvP — **не планируются в принципе** (соло-игра).
- **Решение:** серверный deterministic-replay дня на базе команд — **НЕ делаем.**
  Оверкилл с почти нулевой отдачей: клиентский чит в соло-игре неостановим в принципе,
  никто кроме самого читера не страдает, цена replay (детерминизм, свой PRNG, общее/дублированное
  ядро на сервере, float-дисциплина) несопоставима с угрозой. Противоречит ADR-0001
  (server-authoritative logic = триггер пересмотра).
- **Следствие для симуляции:** покупатель = **чистый state machine, без `ICommand` и без intent-лога.**
  Командная последовательность была нужна только как мост к replay — заказчика больше нет.
  Стек `Game.Commands.ICommand` к этой фиче **не применяем** (он поведенческий и несериализуемый —
  для server-replay не годился бы в любом случае; для real-time симуляции это импеданс-мисматч).
- **Анти-чит (если вообще понадобится) — отложен до релиза**, направление:
  1. сервер владеет балансом gold/reputation, клиентский сейв — кэш (пайплайн Save→сервер уже есть);
  2. bounds-валидация итога дня на сервере (`gold_day ≤ customers × max_book_price × k`, монотонный
     day index, без отрицательных/скачков) — нужен только `SalesDayResult`, не входы;
  3. существующая integrity сейва (SHA256 + монотонный Revision) — первая дешёвая линия.
  Реализуется на фазе Итогов, не в симуляции продажи.

## Открытые вопросы (решить до/в ADR)

1. **Поведение при закрытии/сворачивании приложения** во время дня (suspend/resume vs пересоздание).
   Связано с #7 и #12. Сейчас день пересоздаётся, но поведение на сворачивании (Android onPause)
   не определено.
2. **Детерминизм/тесты для real-time** (tick + `ISalesRandom`, фиксированный `deltaTime`) — желателен
   для тестируемости, но **больше не требуется для server-replay** (replay отброшен). Достаточно
   детерминизма ради юнит-тестов, не ради cross-platform.
3. **Расширение `SalesDayResult`** (обслужено / ушло без покупки / продано active vs passive) — отложено,
   текущий контракт оставляем как есть.
