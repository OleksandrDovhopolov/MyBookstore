# Подготовка

## Назначение

`Подготовка` - стратегическая часть дня. Игрок выбирает, где сегодня поставить лавку, какие книги взять на полку и какие декоры/улучшения использовать. Этот этап должен создавать ощущение осознанного выбора перед продажами.

В MVP этап должен быть простым, но data-driven: лимит книг и лимит декора не должны быть захардкожены. Они являются параметрами прогресса и теоретически могут увеличиваться улучшениями.

## Анализ старта Tiny Bookshop

В референсе подготовка начинается очень мягко: игрок получает маленькую лавку, ограниченное место, один bookcase/небольшой stock и постепенно понимает, что на продажи влияют локация, жанры книг и декор. По обзорам и описаниям, каждый день игрок выбирает место в городе, комплектует полки книгами разных жанров и использует предметы, которые могут усиливать продажи определённых жанров.

Ключевые выводы для нас:

- На старте не давать игроку слишком много решений одновременно.
- Локация должна быть главным решением перед ассортиментом.
- Книги должны выбираться не абстрактно, а под аудиторию локации и утренний модификатор.
- Декор в MVP можно сделать как 1-2 слота с простым бонусом; расширенная система как в рефе остаётся Post-MVP.
- У игрока должно быть ограниченное пространство, иначе подготовка теряет смысл.

## MVP-поведение

Минимальная подготовка состоит из трёх блоков:

1. Выбор локации на карте.
2. Выбор книг на день.
3. Выбор декора/улучшений на день.

Для самого первого прототипа можно иметь только 1 доступную локацию, но UI и модель должны поддерживать список.

## Лимиты

Лимиты должны быть настройками состояния игрока, а не константами:

| Лимит | Базовое значение | Комментарий |
|---|---:|---|
| `dailyBookSlots` | 12-20 | настраиваемое значение; может расти через улучшения лавки |
| `dailyDecorSlots` | 1-2 | настраиваемое значение; может расти через улучшения |

Важно: диапазон 12-20 - это GDD-ориентир, а не фиксированная формула. В коде должен быть один параметр текущей вместимости игрока, например `CurrentBookCapacity`.

## Игровой поток

```text
Open Preparation
  -> Show map
  -> Player selects location
  -> Show inventory books filtered/sorted by relevance
  -> Player selects books up to dailyBookSlots
  -> Show decor/upgrades inventory
  -> Player selects decor up to dailyDecorSlots
  -> Validate setup
  -> Start Sales
```

## Карта и выбор локации

MVP:

- список/карта доступных локаций;
- locked state для будущих локаций;
- краткая аудитория локации;
- активные модификаторы утра;
- expected demand: 2-3 жанра/тега, которые сегодня могут продаваться лучше.

Пример:

```text
Парк
Аудитория: семьи, романтики, casual-читатели
Сегодня: солнечно, прогулочный день
Спрос: kids, romance, light fiction
```

## Выбор книг

Игрок выбирает книги из своего инвентаря. Для MVP достаточно карточек книг:

- название;
- жанр;
- цена;
- теги/тон, если уже добавлены в `BookConfig`;
- индикатор релевантности выбранной локации/событию.

Минимальные состояния:

- доступна;
- выбрана;
- уже на полке;
- не помещается из-за лимита;
- locked/not owned.

Если игрок выбрал меньше минимально допустимого количества, кнопка `Начать продажи` заблокирована или показывает предупреждение. Минимум можно задать отдельным параметром `minDailyBooks`, чтобы не путать с capacity.

## Выбор декора/улучшений

MVP:

- 1-2 слота;
- каждый выбранный предмет даёт простой модификатор;
- пример: `+1 к scoring для genre = adventure`, `+10% шанс фоновой продажи детских книг`.

Post-MVP:

- система как в рефе: декор влияет на жанровые продажи, вводит новые механики, связан с погодой/сезоном/локацией.

## Входные данные

| Источник | Данные | MVP |
|---|---|---|
| Save | owned books, owned decor, unlocked locations, current capacity | да |
| BookConfig | книги, жанры, цены, теги | да |
| LocationConfig | локации, аудитория, demand modifiers | да |
| EventConfig | погода/событие дня | да |
| UpgradeConfig | вместимость полки/слоты декора | позже или stub |

## Выходные данные

```json
{
  "day": 1,
  "phase": "sales",
  "selectedLocationId": "park",
  "selectedBookIds": ["book_001", "book_002"],
  "selectedDecorIds": ["decor_plant"],
  "activeModifiers": ["weather_clear", "event_exam_week"]
}
```

## UI

Рекомендуемая структура:

- верхняя панель: день, золото, активное событие;
- вкладка/секция `Карта`;
- секция `Книги`;
- секция `Декор`;
- нижняя панель: `12/16 книг`, `1/2 декора`, кнопка `Открыть лавку`.

На мобильном UI лучше идти пошагово: `Локация -> Книги -> Декор -> Подтверждение`, чтобы экран не превратился в перегруженную таблицу.

## Validation

Перед стартом продаж:

- выбрана локация;
- выбрано `minDailyBooks <= books <= currentBookCapacity`;
- выбрано `decor <= currentDecorCapacity`;
- все выбранные книги есть в inventory;
- все выбранные декоры есть в inventory;
- нет дубликатов book/decor id.

## Analytics

| Event | Params |
|---|---|
| `preparation_opened` | `day`, `available_locations`, `book_capacity`, `decor_capacity` |
| `location_selected` | `day`, `location_id` |
| `daily_books_confirmed` | `day`, `book_count`, `genre_mix` |
| `daily_decor_confirmed` | `day`, `decor_count`, `decor_ids` |
| `sales_started` | `day`, `location_id`, `book_count`, `decor_count` |

## Save

Сохранять после каждого значимого выбора, но можно debounced:

- `currentPhase = preparation`;
- `selectedLocationId`;
- `selectedBookIds`;
- `selectedDecorIds`.

При рестарте приложения игрок должен вернуться в подготовку с тем же частично собранным setup.

## Acceptance Criteria

- Игрок может выбрать локацию, книги и декор.
- Лимит книг и лимит декора берутся из состояния/конфига, а не из UI-констант.
- Нельзя стартовать продажи без валидного setup.
- Выбранный setup передаётся в `Продажу`.
- При перезапуске во время подготовки выбор восстанавливается.
- Первый прототип работает даже при 1 локации и placeholder-декоре.

## Out Of Scope

- Полноценный drag-and-drop layout полок.
- Продвинутая экономика улучшений.
- Сложная синергия декора.
- Автоматическая оптимизация ассортимента.
- Реальные книги в билде.
