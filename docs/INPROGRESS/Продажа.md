# Продажа

## Назначение

`Продажа` - основная игровая сцена MVP. Здесь проверяется главная гипотеза: интересно ли игроку подбирать книги под людей несколько игровых дней подряд.

Сцена должна связать утренний контекст, подготовленный ассортимент, запросы покупателей и scoring-рекомендации в один короткий, понятный и повторяемый loop.

## Роль в core loop

```text
Утро -> Подготовка -> Продажа -> Итоги
```

На вход сцена получает:

- выбранную локацию;
- выбранные книги на день;
- выбранные декоры/улучшения;
- active event/weather;
- список возможных запросов.

На выходе сцена отдаёт:

- количество покупателей;
- продажи;
- качество рекомендаций;
- золото;
- настроение/репутацию;
- факты для итогов дня.

## MVP-поведение

Продажа состоит из двух типов событий:

1. Фоновая продажа: клиент покупает книгу автоматически по жанровому спросу.
2. Ручная рекомендация: клиент просит подобрать книгу, игрок выбирает книгу с полки.

Для первого playable prototype можно начать только с ручных рекомендаций, но архитектура должна оставить место для фоновых продаж.

## Игровой поток

```text
Open Sales
  -> Spawn next customer/request
  -> Show request text
  -> Player opens shelf
  -> Player selects one available book
  -> Scoring evaluates match
  -> Show result and reason
  -> Apply reward/mood/reputation
  -> Repeat until day timer/customers end
  -> Open Results
```

## Покупательский запрос

Запрос должен быть живой репликой + скрытыми условиями scoring.

Минимальная структура:

```json
{
  "id": "req_001",
  "text": "Хочу научную фантастику про выживание на другой планете, где герой решает проблемы инженерией.",
  "desiredGenres": ["sci-fi"],
  "desiredTags": ["space", "survival", "engineering"],
  "desiredMood": ["smart", "tense", "optimistic"],
  "maxPrice": 80,
  "difficulty": 3,
  "favoriteBookHint": null
}
```

Важно: в билде не использовать реальные современные книги как favorite hint. Вместо `The Martian` использовать вымышленный аналог и проверенные безопасные теги.

## Scoring

Базовая формула из GDD:

| Совпадение | Очки |
|---|---:|
| Жанр | +3 |
| Тег | +2 за тег |
| Тон | +1 |
| Цена подходит | +1 |
| Локация усиливает жанр/тему | +1 |

Порог результата:

| Score | Результат |
|---:|---|
| 0-2 | отказ или недовольная покупка |
| 3-5 | нормальная продажа |
| 6+ | отличная рекомендация, бонус, отзыв |

## Обратная связь

После выбора игрок обязательно должен увидеть причину результата.

Пример:

```text
Отличный выбор.
Совпали: sci-fi, survival, engineering.
Бонус локации: университет усилил научную тему.
```

Без объяснения scoring будет ощущаться случайным, особенно потому что книги вымышленные и игрок не может опереться на реальную читательскую память.

## Состояния книги на полке

| State | Поведение |
|---|---|
| `available` | можно рекомендовать |
| `recommended` | уже была рекомендована в этом запросе |
| `sold_out` | нельзя выбрать, если экземпляры закончились |
| `disabled` | не подходит по технической причине |

Для MVP можно считать, что каждая выбранная книга имеет 1 экземпляр, либо что экземпляры бесконечны в рамках дня. Решение нужно зафиксировать до реализации. Консервативный MVP: 1 экземпляр на выбранную книгу, продажа удаляет её с полки до конца дня.

## Входные данные

| Источник | Данные | MVP |
|---|---|---|
| Preparation result | selectedLocationId, selectedBookIds, selectedDecorIds | да |
| BookConfig | жанр, цена, теги, тон | надо расширить |
| RequestConfig | текст, desired genres/tags/mood, difficulty | надо расширить |
| LocationConfig | demand modifiers | да |
| EventConfig | weather/event modifiers | частично |
| Save | inventory, day state | да |

## UI

Минимальная сцена:

- фон выбранной локации;
- лавка/полка с выбранными книгами;
- текущий покупатель;
- bubble с запросом;
- кнопка/панель выбора книги;
- карточка выбранной книги;
- результат рекомендации;
- счётчики дня: деньги, покупатели, оставшееся время или очередь.

Для MVP можно не делать свободное перемещение. Основной экран должен сразу показывать запрос и доступные книги.

## Завершение дня

Условия завершения:

- кончилась очередь покупателей;
- кончился таймер дня;
- игрок нажал `Закрыть лавку` после минимального числа клиентов.

Для прототипа проще использовать фиксированное количество запросов на день: например 5-8.

## Analytics

| Event | Params |
|---|---|
| `sales_opened` | `day`, `location_id`, `book_count`, `decor_count` |
| `request_shown` | `day`, `request_id`, `difficulty` |
| `book_recommended` | `day`, `request_id`, `book_id`, `score`, `result_tier` |
| `recommendation_reason_shown` | `day`, `request_id`, `matched_genres`, `matched_tags` |
| `sales_completed` | `day`, `customers`, `sales`, `gold`, `excellent_count`, `failed_count` |

## Save

Сохранять:

- `currentPhase = sales`;
- active setup;
- customers served;
- sold books;
- accumulated rewards;
- current request index.

Если игра закрылась во время продажи, MVP-вариант: восстановить сцену на текущем запросе без повторного начисления уже выданных наград.

## Acceptance Criteria

- Сцена принимает setup из `Подготовки`.
- Покупательский запрос отображается игроку.
- Игрок может выбрать только книгу из дневной полки.
- Scoring считает результат по жанрам/тегам/тону/цене/локации.
- Игрок видит понятную причину результата.
- Награды дня аккумулируются и передаются в `Итоги`.
- Нет зависимости от реальных книг или защищённых IP в контенте билда.

## Out Of Scope

- AI-рекомендации.
- Свободное перемещение по локации.
- Сложный customer patience simulation.
- Большая сюжетная система.
- Возвраты книг.
- Полная экономика поставщика книг.
